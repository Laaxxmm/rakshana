## Key architectural decisions

1. **Receipt numbers are allocated inside the donation-creation transaction** via row-locked SQL. If anything inside the transaction throws, the slot is released (verified by a rollback test). Two browsers can never get the same number.
2. **FY rollover auto-creates a new ReceiptSeries** matching the previous FY's prefix/separator/width. Trustees don't need to remember to create one on 1 April. The auto-created series is fully editable in `/settings/organisation` if the operator wants to rename it.
3. **Notify is the same shape as Storage.** Two interfaces, two adapters per channel, factory by env var. Default to console drivers so Phase 2 ships demoable; production turns on Resend + WhatsApp Cloud via env. `[EMAIL]` and `[WHATSAPP]` blocks appear in the dev server log when a donation lands.
4. **Receipt PDF reads from the same Donation row** every time. Regeneration on cancel doesn't preserve a "first version" ‚Äî the prompt asked for an archive path which is captured (`storageKey.donationReceiptArchive`) before the cancel-rewrite, so the original is preserved on disk even though we don't surface it in the UI yet.
5. **Dispatch is fire-and-forget** via `queueMicrotask` after the Server Action response. The donation row + PDF land synchronously; email/WhatsApp drift a few ms later. Notification rows go from `pending` ‚Üí `sentAt`/`errorMessage`. Phase 3 / Phase 5 will move this to a real queue.
6. **Donor profile fetches donations + communications + audit history in parallel.** Single round trip per resource; ~5 queries per page load.

- **Real React Email template** ‚Äî current template is inline HTML. The receipt looks fine in major clients (Gmail web, iOS Mail tested); upgrade to `@react-email/components` when we add more templates.
- **Bell hover preview** ‚Äî same call as Phase 1; the bell is one click from the page.

## Known caveats (record in CODE-HEALTH after this phase)

- **PDFKit silently no-ops on bad images.** Mitigated: `[receipt-pdf] failed to embed ‚Ä¶` warning on load failure. Add a sentry-style alert in Phase 6 deploy.
- **`server-only` package throws outside Next.** Tests use a vitest alias shim; CLI scripts can't easily import server-only modules. Mostly fine ‚Äî the protection is the point.
- **Dispatch uses `queueMicrotask`.** Survives the response but won't survive a process crash. CODE-HEALTH says move to BullMQ + Redis when volume exceeds ~1000 donations/day.
